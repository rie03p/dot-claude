# セキュリティスキャンスキル

コードベースをスキャンし、OWASP Top 10 を中心としたセキュリティ脆弱性を検出する。

## 引数の解析

- パスが指定されたらそのスコープのみ、未指定ならプロジェクト全体を対象にする
- `--focus`: 特定カテゴリに集中（`injection`, `auth`, `xss`, `secrets`, `deps` 等）

## 手順

### 1. プロジェクトの技術スタック特定

package.json、フレームワーク設定ファイル等から使用技術を把握し、スキャン対象を絞る（技術スタックによって検出すべきパターンが異なるため、先に特定する）。

### 2. 自動スキャン

以下のカテゴリを順にチェックする:

#### A. ハードコードされたシークレット

```
検索パターン:
- API キー、トークン、パスワードのリテラル
- Base64エンコードされた認証情報
- プライベートキー
- 接続文字列（DB URL 等）
```

`.env.example` やテストのモック値は除外する（これらは意図的なダミー値であり、誤検出を避けるため）。

#### B. インジェクション脆弱性

- SQL インジェクション（文字列結合によるクエリ構築）
- コマンドインジェクション（ユーザー入力を含むシェル実行）
- NoSQL インジェクション
- テンプレートインジェクション

#### C. XSS / 出力エスケープ

- `dangerouslySetInnerHTML`、`innerHTML` の使用
- ユーザー入力の未サニタイズ出力
- URL パラメータの直接埋め込み

#### D. 認証・認可

- 認証バイパスの可能性
- 不適切な権限チェック
- セッション管理の問題
- CORS 設定の甘さ

#### E. 設定・インフラ

- デバッグモードの本番有効化
- 不適切な HTTPS/TLS 設定
- 過度に寛容な CORS
- セキュリティヘッダーの欠落

#### F. 依存パッケージ

- 既知の脆弱性を持つバージョンの使用（`/dependency-check` と連携する）

### 3. レポート出力

```
## セキュリティスキャンレポート

### 対象
- スキャン範囲: ...
- 技術スタック: ...

### 検出事項

#### Critical
| # | ファイル:行 | カテゴリ | 内容 | 推奨対応 |
|---|---|---|---|---|

#### Warning
| # | ファイル:行 | カテゴリ | 内容 | 推奨対応 |
|---|---|---|---|---|

#### Info（ベストプラクティス推奨）
| # | ファイル:行 | カテゴリ | 内容 |
|---|---|---|---|

### サマリー
- Critical: X件 / Warning: Y件 / Info: Z件
```

### 4. 修正支援

ユーザーが求めた場合、個別の脆弱性について:
- 具体的な修正コードを提案する
- 修正の実装を実行する

## 使用例

**例1**: ユーザーが「リリース前にセキュリティチェックしたい」と言った場合
→ プロジェクト全体をスキャンし、全カテゴリのレポートを出力する。

**例2**: ユーザーが「src/api/ にSQLインジェクションの問題がないか確認して」と言った場合
→ `--focus injection` で src/api/ ディレクトリのみを対象にスキャンする。

**例3**: ユーザーが「.env がコミットされてないか心配」と言った場合
→ シークレット検出にフォーカスし、ハードコードされた認証情報や .env ファイルの git 管理状況を確認する。
