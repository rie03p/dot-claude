# テスト仕様書作成スキル

指定された Linear サブチケット（テスト仕様書作成タスク）の内容に基づき、親Issue・関連Issueの情報も参照してテスト仕様を設計し、TSVファイルを出力する。

## 使用例

**例1**: 「GLU-456のテスト仕様書を作って」
→ GLU-456 とその親Issue・関連Issueを取得し、因子と水準を整理してテストケースを設計、TSVファイルとして出力する。

**例2**: 「/create-test-spec GLU-789 --tsv-template ~/templates/test.tsv」
→ 指定されたTSVテンプレートの列構成に従ってテスト仕様書を作成する。

**例3**: 「このチケットのテストケースを洗い出して」（会話中にチケットIDが出ている場合）
→ 会話中のチケットIDを使ってテスト仕様書を作成する。

## 引数の解析

`$ARGUMENTS` からチケットID（例: `PROJ-1234`）を読み取る。未指定ならユーザーに聞く。

オプション引数:
- `--tsv-template <path>`: TSV列テンプレートファイル。指定がなければデフォルトテンプレート（後述）を使う。

## 手順

### 1. チケット情報の取得（親Issue・関連Issueを含む）

`linear api` で対象チケットを起点に、親Issue・子Issue・関連Issueを取得する。`searchIssues(term: ..., first: 1)` を使う。

```bash
linear api '
query {
  searchIssues(term: "<チケットID>", first: 1) {
    nodes {
      id
      identifier
      title
      description
      state { name type }
      team { key id }
      parent {
        id
        identifier
        title
        description
      }
      children {
        nodes {
          id
          identifier
          title
          description
          state { name type }
        }
      }
      relations {
        nodes {
          id
          type
          issue {
            id
            identifier
            title
            description
            state { name type }
          }
          relatedIssue {
            id
            identifier
            title
            description
            state { name type }
          }
        }
      }
    }
  }
}
'
```

取得した情報を表示:
- 対象チケットのタイトル・description
- 親Issueのタイトル・要件・制約
- 子Issue（存在する場合）の観点・影響範囲
- 関連Issue（relation）の観点・依存関係・類似不具合

テスト観点を整理するときは、対象チケットだけでなく、親Issue・関連Issueの要件/既知制約/影響範囲を統合して判断する（対象チケット単体では見落としやすい影響範囲や制約が親・関連Issueに記載されていることが多いため）。

### 2. テスト設計（因子と水準の整理）

チケット description に因子水準設計が既に記載されている場合はそれを使う（plan-ticket スキルで事前に設計済みのことがあるため、二重作業を避ける）。記載がない場合は以下の手順で設計する。

テストの基本方針: **触った箇所がバグっていないことを確認する**。機能追加を含む場合は新機能の動作確認も含める。

#### 因子と水準の考え方

- **因子（Factor）**: テスト対象の変数・条件（例: ユーザー種別、入力値、状態）
- **水準（Level）**: 各因子が取りうる値・状態（例: 管理者/一般ユーザー、有効値/無効値/境界値）

#### 整理手順

1. 修正・追加した箇所から「何が変わると振る舞いが変わるか」を抽出 → 因子
2. 各因子について「どんな値・状態があるか」を列挙 → 水準
3. **フル掛け算のケース数を明示**: A(n) × B(m) × ... = XX ケース
4. 省略できる組み合わせを判断し、実際のケース数を決定
5. エッジケース・境界値を追加

#### 組み合わせの省略判断

フル掛け算したケースを以下の基準で削減し、省略した理由を記録する（後から「なぜこのケースを飛ばしたのか」が追えないと、テスト漏れの判断ができなくなるため）:

| 省略できる条件 | 例 |
|---|---|
| コードパスが共通（別の因子水準でも同じ処理） | 状態A/状態Bで同じコンポーネントを使う → 一方のみ確認 |
| テスト環境で再現困難 | 特定の状態を手動で再現できない → 別の方法で代替 |
| 他ケースの実データに自然に混在する | 大量データには各種バリエーションが含まれる |
| 別の因子の確認に完全に包含される | 正常系が通れば必ず通るケース |

#### テストケースの構成

以下の形式でテストケースを整理する:

**テストケース一覧:**

| # | （因子A） | （因子B） | ... | 確認内容 |
|---|-----------|-----------|-----|---------|
| TC-1 | ... | ... | ... | ... |

**各テストケースの確認手順と確認項目:**（TC ごとに操作手順と確認チェックリストを記述）

**省略したケースとその理由:**

| 省略したケース | 理由 |
|---|---|
| （因子A の水準X） × （因子B の水準Y） | （省略理由。コードパス共通/再現困難/混在するなど） |

整理したテストケース一覧をユーザーに見せ、追加・修正を確認してから次のステップへ進む。

### 3. TSVファイルの作成

ステップ2で整理したテストケースを TSV ファイルとして書き出す。

- チャット上のコードブロックからコピーするとタブが失われるため、ファイル経由で渡す
- 出力先: `~/Downloads/test-spec-<チケットID>.tsv`（Write ツールで作成）
- 既存ファイルがある場合は上書き前にユーザーへ確認する

#### TSVテンプレート

`--tsv-template` 引数でテンプレートファイルが指定されている場合、そのファイルの列構成に従う。

指定がない場合は以下のデフォルトテンプレートを使う:

| 列 | 列名 | 内容 |
|---|---|---|
| 1 | No | 連番 |
| 2 | デグレチェック(1/0) | コアの正常系・デグレ確認は1、エッジケースは0 |
| 3 | テスト大項目 | 機能名 |
| 4 | テスト中項目 | シナリオ説明 |
| 5 | テスト小項目 | 具体的な確認内容 |
| 6 | 前提条件 | アカウント種別・状態など |
| 7 | テスト手順 | ①②③形式で記述 |
| 8 | 期待値 | ・箇条書き形式で記述 |
| 9 | テスト結果 | 空欄（テスター記入） |
| 10 | テスト結果詳細 | 空欄 |
| 11 | テスト日 | 空欄 |
| 12 | テスター | 空欄 |
| 13 | 備考 | 空欄または補足 |

出力形式（ヘッダー行 + データ行のタブ区切り）:

- 前提条件・テスト手順・期待値はセル内で改行したいため、**ダブルクォートで囲んで実際の改行文字を含める**
- テスト手順は `①②③` の連番形式、期待値は `・` 箇条書きで各項目を改行して記述

```
No	デグレチェック(1/0)	テスト大項目	テスト中項目	テスト小項目	前提条件	テスト手順	期待値	テスト結果	テスト結果詳細	テスト日	テスター	備考
1	1	（大項目）	（中項目）	（小項目）	"条件1
条件2"	"①手順1
②手順2
③手順3"	"・期待値1
・期待値2"
```

**デグレチェックの判断基準:**
- `1`: コアの正常系（新機能の主要フロー）・既存機能のデグレ確認
- `0`: バリエーション確認・エッジケース・既存動作への影響が薄いケース

### 5. Linearチケットへの成果物記載

作成したTSVファイルパスと要約をLinearチケットのdescriptionに追記する。descriptionには **因子と水準の表・フル掛け算のケース数・省略したケースと理由** を含める。

文字列のエスケープ問題を避けるため、`--variables-json` を使って Python 経由で更新する:

```bash
python3 << 'EOF'
import subprocess
import json

description = """<既存のdescription>

## テスト設計

### 因子と水準

| 因子 | 水準 |
|---|---|
| A: <因子A名> | <水準1> / <水準2> |
| B: <因子B名> | <水準1> / <水準2> / <水準3> |

フル掛け算: A(n) × B(m) × ... = XXケース → 実施: Yケース

### 省略したケースとその理由

| 省略したケース | 理由 |
|---|---|
| <因子A の水準X> × <因子B の水準Y> | <省略理由> |

## テスト仕様書（TSV）

- 出力ファイル: ~/Downloads/test-spec-<チケットID>.tsv
- ケース数: <件数>（デグレチェック対象: <件数>件、対象外: <件数>件）
- 親Issue/関連Issue反映: <要約>"""

mutation = """
mutation($id: String!, $description: String!) {
  issueUpdate(id: $id, input: { description: $description }) {
    success
    issue { identifier title }
  }
}
"""

variables = {
    "id": "<チケットの id>",
    "description": description
}

result = subprocess.run(
    ["linear", "api", mutation, "--variables-json", json.dumps(variables)],
    capture_output=True,
    text=True
)
print(result.stdout)
if result.stderr:
    print("STDERR:", result.stderr)
EOF
```

### 6. 完了報告

以下をユーザーに報告する:
- 作成したTSVファイルパス
- テストケース数・概要
- 親Issue・関連Issueを参照して反映した観点
- Linearチケットの更新状況
