# リファクタリングスキル

指定されたコードを分析し、リファクタリング案を提示して実行する。

## 引数の解析

`$ARGUMENTS` からファイルパスまたは関数名を読み取る。未指定ならユーザーに聞く。

## 手順

### 1. 対象コードの分析

対象ファイル/関数を読み込み、以下を分析する:

- コードの責務と依存関係
- 問題点の特定:
  - 重複コード
  - 長すぎる関数（目安: 50行以上。長い関数はテストしにくく、バグの温床になりやすい）
  - 深いネスト（3階層以上。認知負荷が急激に上がるため）
  - 不明瞭な命名
  - 不要な複雑さ
  - 型安全性の問題
- 呼び出し元/呼び出し先の把握（影響範囲を先に把握しないと、リファクタリングで予期せぬ箇所が壊れるため）

### 2. リファクタリング案の提示

分析結果をもとに、具体的な改善案を優先度付きで提示する:

```
## リファクタリング案

### 高優先度
1. [具体的な変更内容] — 理由: ...
2. ...

### 中優先度
1. ...

### 影響範囲
- 変更されるファイル: ...
- テストへの影響: ...
```

ユーザーの承認を得てから次へ進む。一括で進めるとロールバックが困難になるため、承認プロセスを省略しない。

### 3. 段階的な実行

承認された項目を1つずつ実行する（一度に全部変更すると、どの変更が問題を起こしたか特定できなくなるため）:

1. 変更を適用
2. 既存テストがあれば実行して壊れていないことを確認
3. 型チェック/リントがあれば実行
4. 次の変更へ

### 4. 完了報告

- 変更箇所のサマリー
- Before/After の比較（主要な変更のみ）
- テスト結果
- 追加で検討すべきリファクタリング（あれば）

## 使用例

**例1**: ユーザーが「src/utils/format.ts が長すぎるから整理して」と言った場合
→ ファイルを読み込み、責務ごとに関数を分割する案を提示。承認後に段階的に実行する。

**例2**: ユーザーが「handleSubmit 関数のネストが深くて読めない」と言った場合
→ 該当関数を分析し、早期リターンやヘルパー関数への抽出を提案する。

**例3**: ユーザーが「このディレクトリのコード品質を上げたい」と言った場合
→ ディレクトリ内のファイルを走査し、改善すべき箇所を優先度付きでリストアップする。
