# チケット計画スキル

指定された Linear チケットの情報を取得し、description を充実させ、必要な Sub Issue（実装・テスト仕様書・テスト実施）を作成する。

## 使用例

**例1**: 「GLU-456を計画して」
→ GLU-456 のチケット情報を取得し、PMモードか実装者モードかを確認後、descriptionを整理してサブチケットを作成する。

**例2**: 「このチケットを分解したい。認証機能の追加で、OAuth2対応が必要。ゴールはGoogle/GitHubログインの実装」
→ 会話中に十分な情報があるのでインタビューをスキップし、そのまま description 更新とサブチケット作成に進む。

**例3**: 「GLU-789のサブタスクを作って --labels Test:abc123,App:def456」
→ チケット情報を取得し、サブチケットを作成、指定されたラベルを付与する。

## 引数の解析

`$ARGUMENTS` からチケットID（例: `PROJ-1234`）を読み取る。未指定ならユーザーに聞く。

オプション引数:
- `--labels <ラベル名:ID,...>`: Sub Issue やチケットに付与するラベルのマッピング（例: `Test:abc123,App:def456`）。指定がなければラベル付与をスキップする。

## 手順

### 1. モード選択と会話コンテキストの確認

まず、AskUserQuestion でモードを確認する:

- **PMモード（デフォルト）**: チケットの計画のみ行い、実装は他の人に任せる。コードベース調査は行わない。実装サブチケットにも技術詳細は書かない。
- **実装者モード**: 自分で実装する前提で計画する。コードベース調査を行い、実装サブチケットに技術情報（関連ファイル・変更方針）を含める。

次に、スキル呼び出し前の会話を確認し、ユーザーが既に以下の情報を提供しているか判定する:

- **目的/背景**: なぜこの機能・修正が必要か
- **要件/詳細**: 具体的にどういう振る舞いを期待するか
- **ゴール/ノンゴール**: スコープの明確化
- **備考**: その他の制約や注意事項

**十分な情報がある場合** → インタビュー（手順4）をスキップし、手順5に進む
**情報が不足している場合** → 不足部分のみ手順4で質問する

### 2. チケット情報の取得

`linear api` で対象チケットの詳細を取得する。`searchIssues(term: ..., first: 1)` を使う。

```bash
linear api '
query {
  searchIssues(term: "<チケットID>", first: 1) {
    nodes {
      id
      identifier
      title
      description
      priority
      estimate
      state { name type }
      assignee { name }
      team { key id }
      parent { identifier title }
      children {
        nodes {
          identifier
          title
          state { name type }
          assignee { name }
          estimate
        }
      }
      labels { nodes { name } }
      comments(first: 5, orderBy: createdAt) {
        nodes {
          body
          user { name }
          createdAt
        }
      }
    }
  }
}
'
```

取得した情報をユーザーに表示する:
- タイトル、現在の description、ステータス、担当者、既存の子チケット
- コメントがあれば直近のものを表示

### 3. コードベース調査（実装者モードのみ）

**PMモードではこの手順をスキップする。**

実装者モードの場合、以下を行う:
- Grep / Glob でキーワード検索（関数名、コンポーネント名、API名など）
- 関連ファイルを特定して読み込む
- 既存の実装パターン、影響範囲、依存関係を把握する

### 4. インタビュー（情報が不足している場合のみ）

手順1で判定した不足部分のみを質問する。AskUserQuestion を使って 1〜3 問ずつ段階的に聞く。

**全ての情報が揃っている場合はこの手順をスキップする。**

#### 聞くべき観点（不足している場合のみ）

**背景と目的:**
- この機能/修正の背景は？（なぜ必要か）
- ユーザーにとってどういう価値があるか？

**要件の明確化:**
- 具体的にどういう振る舞いを期待するか？
- ゴール・ノンゴールはあるか？
- エッジケースや制約条件はあるか？

**実装方針の確認（実装者モードのみ）:**
- コードベース調査から見えた実装アプローチについて、ユーザーの意向を確認
- テストで特に検証すべき観点はあるか？

※ 技術情報は**親チケットには書かず、実装サブチケットの description に使う**。

### 5. 親チケット description の更新

会話コンテキスト + インタビュー結果を元に、親チケットの description を更新する。

親チケットにはPM視点・現象ベースの情報のみを書く。技術的な詳細（ファイル名、行番号、関数名、コードスニペット、変更方針、実装アプローチ、テスト設計の詳細）は実装サブチケットに書く。

理由: 親チケットはPM・デザイナー・QAなど非エンジニアも参照する。「何が起きているか」「何を達成したいか」が伝わればよく、「どう直すか」はエンジニア向けの実装サブチケットに分離したほうが情報の見通しがよくなる。

```markdown
## 概要
（何が起きているか / 何をしたいか。1〜2文で。現象ベースで書く）

## 詳細
（具体的な仕様・振る舞いの説明。再現手順があればここに含める）

## ゴール
（このチケットが完了した状態。ユーザー視点で書く）

## ノンゴール
（今回やらないこと）
```

必要に応じて以下のセクションも追加する:
- **背景** — 経緯や動機が複雑な場合
- **要件** — チェックリスト形式が適切な場合
- **影響範囲** — どのユーザー・機能に影響するか

更新前にユーザーにプレビューを見せ、承認を得てから更新する。

文字列のエスケープ問題を避けるため、`--variables-json` を使って Python 経由で更新する:

```bash
python3 << 'EOF'
import subprocess
import json

description = """（新しい description）"""

mutation = """
mutation($id: String!, $description: String!) {
  issueUpdate(id: $id, input: { description: $description }) {
    success
    issue { identifier title }
  }
}
"""

variables = {
    "id": "<チケットの id>",
    "description": description
}

result = subprocess.run(
    ["linear", "api", mutation, "--variables-json", json.dumps(variables)],
    capture_output=True,
    text=True
)
print(result.stdout)
if result.stderr:
    print("STDERR:", result.stderr)
EOF
```

### 6. Sub Issue の作成

チケットの内容に基づき、以下の観点で Sub Issue を計画する:

#### 必須タスク
1. **実装** — 実際のコード変更。規模が大きい場合はさらに分割（調査、フロント実装、バックエンド実装など）
2. **テスト仕様書作成** — テスト観点・テストケースの設計（下記の因子水準設計を含む）
3. **テスト実施** — テスト仕様書に基づくテスト実行

#### 必要に応じて追加
- **調査** — 不明点が多い場合に事前調査タスク
- **デザイン作成** — UI変更を伴う場合
- **ドキュメント作成** — 手順書や仕様書が必要な場合
- **データ移行/修正** — 既存データへの影響がある場合

各 Sub Issue には、そのタスクを担当する人が必要とする情報だけを書く。親チケットの内容を繰り返さない（重複があると更新時に不整合が生じるため）。

**実装タスク（PMモード）:**
```markdown
## やること
- 具体的な変更内容1
- 具体的な変更内容2
- ...

## 完了条件
- [ ] 条件1
- [ ] 条件2
```

**実装タスク（実装者モード）:**
```markdown
## やること
- 具体的な変更内容1（ファイル名・関数名レベルで）
- 具体的な変更内容2
- ...

## 技術的な調査メモ
- 関連ファイル: ...
- 既存実装の概要: ...
- 変更方針: ...

## 完了条件
- [ ] 条件1
- [ ] 条件2
```

**テスト仕様書作成タスク:**

plan-ticket の時点では因子・水準・省略理由までを書く。テストケース（手順・確認項目）は実装後に create-test-spec スキルで具体化する。実装前には影響範囲が正確にわからないため、ここでは「何を観点として見るか」の骨格を示すに留める。

```markdown
## 目的
（触った箇所がバグっていないことを確認する、など）

## 因子と水準

| 因子 | 水準 |
|---|---|
| A: （因子名） | A1: ... / A2: ... / ... |
| B: （因子名） | B1: ... / B2: ... / ... |

フルの掛け算だと A(n) × B(m) × ... = XX ケースになるが、以下の理由で YY ケースに絞る見込み。

## 省略できそうなケースとその理由

| 省略候補 | 理由 |
|---|---|
| （因子A の水準X） × （因子B の水準Y） | （省略理由） |

## 備考
- テストケース（手順・確認項目）は実装完了後に具体化する
- 実装で判明した影響範囲に応じて因子・水準を追加・修正する
```

因子水準の設計方針:
- **因子の抽出**: 修正箇所に関係する変数・条件（ユーザー状態、コンテンツ種別、データ量など）
- **フル掛け算の計算**: A(n) x B(m) x ... = 総ケース数を明示する（全体像を把握してから削減判断をするため）
- **省略の判断基準**（省略する場合は理由を書く。後から「なぜこのケースを飛ばしたか」を追えるようにするため）:
  - コードパスが共通なら一方のみ（例: 状態A/状態Bで同じコンポーネントを使う）
  - テスト環境で再現が困難（例: 特定の条件下でしか発生しない状態）
  - 他ケースの実データに自然に混在する

**テスト実施タスク:**
```markdown
## テスト仕様書
（テスト仕様書作成タスク完了後にURLを記載）

## 完了条件
- [ ] テスト仕様書の全ケースを実施済み
- [ ] 未解決のバグがない（またはチケット起票済み）
```

Sub Issue の一覧をユーザーに提示し、承認を得てから作成する。

`--labels` 引数が指定されている場合、該当するラベルを付与する。例えばテスト実施サブチケットに「Test」ラベルを付けたい場合、`--labels Test:<label-id>` のように指定する。作成後に `issueAddLabel` で追加する。

Sub Issue の作成も `--variables-json` を使って安全にエスケープする:

```bash
python3 << 'EOF'
import subprocess
import json

title = "<タイトル>"
description = """<description>"""

mutation = """
mutation($title: String!, $description: String!, $teamId: String!, $parentId: String!) {
  issueCreate(input: { title: $title, description: $description, teamId: $teamId, parentId: $parentId }) {
    success
    issue { identifier title }
  }
}
"""

variables = {
    "title": title,
    "description": description,
    "teamId": "<teamId>",
    "parentId": "<親チケットの id>"
}

result = subprocess.run(
    ["linear", "api", mutation, "--variables-json", json.dumps(variables)],
    capture_output=True,
    text=True
)
print(result.stdout)
if result.stderr:
    print("STDERR:", result.stderr)
EOF
```

※ `teamId` は Step 2 で取得したチケットの `team { id }` から使う。

### 7. 完了報告

最後に以下をユーザーに報告する:
- 更新した description の要約
- 作成した Sub Issue の一覧（identifier + タイトル）
- 次のアクションの提案
