## Context

- Current git status: !`git status`
- Current git diff (staged and unstaged changes): !`git diff HEAD`
- Current branch: !`git branch --show-current`
- Recent commits: !`git log --oneline -10`

## タスク

ステージされたファイルから変更内容を分析し、3つのコミットメッセージ候補を生成してユーザーに選択させる。

1. **変更の分析**: diff と git status から変更の種類・影響範囲・目的を把握する
   - ステージ済みの変更がない場合は、unstaged changes や untracked files を確認し、何をステージするかユーザーに提案する。変更自体がなければ「コミットする変更がありません」と伝えて終了する
2. **コミット内容の確認**: 変更内容を確認し、違う作業種類の変更が混在していて分離可能であれば警告を出し、分割を促す（例: バグ修正と機能追加が混在している場合など）。1コミット1関心事にすると、後から revert やcherry-pick がしやすくなるため
3. **タイトル候補の生成**: 異なる視点で3つのタイトル候補を作成する
   - Conventional Commits 形式（feat/fix/docs/refactor/chore 等）
   - モジュール名（frontend, backend, api 等。パッケージ名やディレクトリ名を利用）を `[xxx]` の形式でプレフィックスにする
   - 日本語で記述（プレフィックスは英語）
   - 各候補は異なる詳細度や視点を提供する（技術的詳細重視、ビジネス価値重視、シンプル表現）
   - 例: `feat: [frontend] ユーザープロフィール画面のUI改善`、`fix: [api] 認証エラーの修正`
4. **ユーザー選択**: AskUserQuestion で3候補を提示し選択を受ける
5. **本文を作成**: 選択されたタイトルに基づいて変更内容を説明する本文を生成する。変更内容の概要、影響範囲、新たに出てきた TODO を含める
6. **コミット実行**: 必要に応じて git add し、選択されたメッセージでコミットする

## 使用例

**例1**: ユーザーが「コミットして」と言った場合
→ git status と diff を確認し、ステージ済みの変更から3つのコミットメッセージ候補を提示する。

**例2**: ユーザーが「変更をまとめて」と言ったが、ステージ済みファイルがない場合
→ unstaged changes と untracked files を一覧表示し、「これらをステージしますか？」と提案する。

**例3**: バグ修正と新機能のコードが混在してステージされている場合
→ 「バグ修正と機能追加が混在しています。分けてコミットしますか？」と確認する。

## 制約

- Claude co-authorship フッターは不要（プロジェクトのコミットログの統一性を保つため）
- メッセージは日本語（プレフィックスのみ英語）
